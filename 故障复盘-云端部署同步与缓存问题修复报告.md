# ChickenNoteLM 云端部署故障复盘与修复报告

## 1. 背景

项目最初是单机本地笔记应用，数据以本地文件为主。迁移到云端后，需要支持：

- 多用户访问
- 浏览器本地缓存（IndexedDB/localStorage）与服务器文件双向同步
- 登录态隔离（按用户分目录）

在此过程中，前后端同步逻辑与缓存策略逐步叠加，最终出现了高频报错与数据显示异常。

---

## 2. 线上现象（用户可见）

### 2.1 初期核心报错

- 控制台反复出现：
  - `GET /api/auth/me 401 (UNAUTHORIZED)`
  - `POST /api/sync/notes-events net::ERR_INSUFFICIENT_RESOURCES`
  - `同步到本地文件失败 TypeError: Failed to fetch`
- 页面可短暂使用，但会频繁出现同步失败。

### 2.2 二次问题（修复后暴露）

- 编辑并保存后，左侧列表摘要会更新。
- 但刷新页面或切换到其他笔记再切回时，编辑区正文变空。

---

## 3. 影响范围

- 高并发无效请求导致浏览器资源耗尽（`ERR_INSUFFICIENT_RESOURCES`）。
- 同步行为不稳定，可能导致用户误以为“保存失败”。
- 预览/编辑视图切换逻辑会覆盖正文显示，造成“数据丢失错觉”。
- Service Worker 静态缓存策略导致“已修复代码未生效”的错觉。

---

## 4. 根因分析

## 4.1 根因 A：前端保存链路递归触发（请求风暴）

### 问题链路

1. `state.js` 的 `saveDataToStorage()` 遍历 `notes` 调用 `dataService.saveNote()`
2. `data-service.js` 的 `saveNote()` 内又调用 `saveToLocalStorage()`
3. `saveToLocalStorage()` 回调 `saveDataToLocalStorage()` -> 再进入 `saveDataToStorage()`
4. 形成递归，触发大量 `fetch` 请求（含 `/api/sync/notes-events`）

### 结果

- 请求风暴
- 浏览器资源耗尽
- 同步失败噪音大量出现

---

## 4.2 根因 B：增量同步后端接口参数/路径错误

后端 `chickennotelm_server.py` 中存在多处参数使用不一致与路径引用错误，例如：

- `load_sync_state(...)` / `save_sync_state(...)` 调用参数错位
- `get_note_modified_at(...)` 调用缺少 `note_dir`
- `/api/events` 读取了全局 `EVENT_DIR`，而非当前用户目录

### 结果

- `/api/sync/pull`、`/api/sync/push` 行为不稳定，易出现“拉不到/推不准”

---

## 4.3 根因 C：编辑器视图恢复逻辑误覆盖内容

`notes.js` 的 `renderEditorView()` 在编辑模式下也会恢复 `_originalMarkdownContent`。  
当该变量为空或过期时，会把编辑器当前内容覆盖为空，表现为“切回来正文没了”。

---

## 4.4 根因 D：Service Worker 缓存策略导致旧 JS 长驻

`sw.js` 早期是固定 cache version + 缓存优先，导致用户可能长期命中旧脚本。  
即使服务器已修复，前端仍可能运行旧逻辑。

---

## 5. 已实施修复（生产环境）

生产路径：`/var/www/chickennoteLM`

### 5.1 前端保存链路修复

文件：`js/state.js`

- `saveDataToStorage()` 不再调用 `dataService.saveNote()`（避免递归）
- 改为直接写 IndexedDB：`window.dataService.db.addNote(note)`
- `saveDataToLocalStorage()` 改为仅写 localStorage，不再触发完整保存链路
- `syncNotesAndEventsToServer()` 增加登录态保护：
  - 登录遮罩显示时直接跳过
  - 遇到 `401` 静默返回（不抛错误噪音）

文件：`js/data-service.js`

- `saveNote/updateNote/deleteNote` 移除 `saveToLocalStorage()` 调用，切断循环
- `pull/push` 请求补充 `credentials: 'include'`
- `pull/push` 收到 `401` 时直接返回，避免异常连锁

---

### 5.2 后端增量同步修复

文件：`chickennotelm_server.py`

- 修正 `load_sync_state` / `save_sync_state` 的参数传递与目录初始化
- 修正 `get_note_modified_at(note_dir, note_id)` 的调用
- 修正 `/api/events` 为按用户目录读取事件文件

---

### 5.3 编辑区正文变空修复

文件：`js/notes.js`

- 新增 `_lastEditorView`
- 仅当“刚从预览模式切回编辑模式”时才恢复 `_originalMarkdownContent`
- 避免普通切换笔记/初始化时误覆盖编辑器正文

---

### 5.4 Service Worker 缓存策略修复

文件：`sw.js`

- cache version 升级：`v1 -> v2 -> v3`
- 保留缓存命中优势，同时增加“命中后后台更新缓存”
- 降低旧脚本长期驻留风险

---

## 6. 验证结果

已在服务器完成接口与运行验证：

- `POST /api/auth/login` -> 200
- `GET /api/auth/me` -> 200（登录后）
- `POST /api/sync/pull` -> 200
- `POST /api/sync/push` -> 200
- 页面验证：编辑保存 -> 切换笔记 -> 刷新页面 -> 正文仍可正常显示

说明：未登录时 `GET /api/auth/me` 返回 `401` 属于预期行为，不是故障本身。

## 7. 给后续开发者的规避建议（重点）

### 7.1 保存链路设计原则

- `state 层` 与 `data-service 层` 必须单向调用，禁止双向回调形成环
- localStorage、IndexedDB、远端同步要有明确“主写入口”
- 建议引入“写入来源标记”（例如 `source: ui|sync|hydrate`）防止回流

### 7.2 同步策略建议

- 保存操作使用防抖（如 800~1500ms）
- 远端同步使用队列/节流，避免每次输入都发请求
- 对 401、网络离线、5xx 分层处理，避免噪音日志淹没真实故障

### 7.3 鉴权与初始化建议

- 首屏先确认登录态，再启动同步循环
- 未登录禁止触发任何会写服务器的请求
- 同步 API 统一携带 `credentials: 'include'`

### 7.4 Service Worker 发布建议

- 每次关键前端改动必须升级 cache version
- 尽量采用“stale-while-revalidate”式策略，避免缓存永久陈旧
- 发布说明中明确提醒强刷（`Ctrl+F5`）或自动触发更新提示

### 7.5 代码质量建议

- 为同步核心链路补充最小集成测试：
  - 保存后刷新仍可读
  - 切换笔记不丢正文
  - 401 状态下不触发写请求风暴
- 对关键状态变更加可控日志（可开关）

## 9. 结论

本次问题的本质是：**本地优先缓存、同步队列、登录鉴权、视图状态恢复、SW 缓存** 五条链路耦合后缺少边界，导致回调环和状态覆盖。  
通过“切断递归、修正同步接口、修复视图恢复条件、升级缓存策略”后，功能已恢复稳定。

后续如继续演进，建议优先做“同步状态机化 + 写入队列化 + 可观测性补齐”，避免同类问题再次出现。
